# WF_DSC_PROJECT_INTAKE
**D.S.C — Department of Systems Cultivation & Project Creation**
**Workflow: From Raw Idea to Sanctioned Project**

---

## WHAT THIS WORKFLOW IS

The gateway sequence for every project that enters the Pandora OS. No project moves into active build without passing through this workflow. It applies the alchemical lifecycle as a practical evaluation and design process — taking the raw material of an idea through purification, illumination, and integration before committing resources to it.

This workflow protects the OS from three failure modes that kill most sovereign builders:
- Chasing every idea without finishing anything
- Building the wrong thing with full effort
- Starting without a clear enough picture to know when done

---

## TRIGGER

Any of the following activates this workflow:

- A new idea that demands attention and wants to become a project
- An opportunity that appears from outside and needs evaluation
- An existing project that has lost direction and needs to be re-evaluated from the beginning
- A request from another department to initiate something new

---

## INPUT REQUIRED

```
IDEA/OPPORTUNITY: What is it, stated plainly in one sentence?
SOURCE: Where did this come from?
        (internal inspiration / external opportunity / department need)
URGENCY SIGNAL: Is there a time element or is this open?
INITIAL ENERGY LEVEL: On a scale of 1-10, how much pull does this have?
DEPARTMENT ORIGIN: Which department does this primarily belong to?
```

## THE ALCHEMICAL LIFECYCLE — AS EXECUTABLE PROCESS

The four stages are not metaphor here. They are a practical evaluation and development sequence. Each stage has a gate. The idea does not proceed until the gate is passed.

```
NIGREDO     → Raw material. What actually is this?
ALBEDO      → Purification. Does it deserve to exist?
CITRINITAS  → Illumination. What does it actually require?
RUBEDO      → Integration. How does it enter the OS?
```

## THE WORKFLOW — STEP BY STEP

### STEP 1 — NIGREDO: RAW MATERIAL EXTRACTION
**See the idea clearly before evaluating it.**

Most ideas arrive distorted — inflated by excitement or deflated by doubt. This step strips both distortions and looks at what is actually there.

```
ACTION — Answer each of these without editing:

1. PLAIN STATEMENT
   Describe this idea in one sentence.
   Not the vision — the actual thing.
   "A YouTube channel about sovereign living"
   not "a media empire that changes consciousness"

2. ORIGIN TRACE
   Where did this come from?
   A problem you experienced?
   Something you consumed that sparked it?
   A gap you noticed in the market?
   An instruction from inner intelligence?
   Trace it honestly — origin reveals motivation.

3. WHAT PROBLEM DOES IT SOLVE?
   For you? For others?
   If it solves no problem — what does it create or express?
   There must be a genuine answer here.

4. RAW OBSTACLES
   Without filtering — what stands between now and this existing?
   List every obstacle that comes to mind immediately.
   Do not problem-solve yet. Just name them.

5. SIMILAR THINGS THAT EXIST
   What already exists that is similar?
   This is not a reason not to proceed —
   it is information about the landscape.

OUTPUT: Nigredo document — raw, unedited answers to all five.
        No polish. This is the ore, not the refined metal.
```

**NIGREDO GATE:**
Does enough raw material exist to justify purification?
If the plain statement cannot be written in one sentence —
the idea is not ready. Let it develop and return.

---

### STEP 2 — ALBEDO: PURIFICATION
**Test the idea against the sovereign standard.**

This step asks the hard questions. An idea that cannot survive these questions should not receive resources. An idea that passes them has earned its place.

```
ACTION — The five purification tests:

TEST 1 — ALIGNMENT
Does this serve the Pandora Empire mission?
Does it connect to at least one of the three Triple Helix pillars
(STIS / RWIS / PNC-AI) or to the long-term sovereign estate vision?
If it connects to nothing — why is it here?

TEST 2 — SOVEREIGNTY CHECK
Does this expand or contract sovereignty?
   Expands: generates income, builds skill, produces assets,
            increases freedom, serves others genuinely
   Contracts: creates dependency, dilutes focus,
              serves someone else's agenda at sovereign cost
Be honest. Some opportunities look like expansion
and are actually contraction in disguise.

TEST 3 — RESOURCE REALITY
What does this actually cost?
   Time: hours per week, for how long?
   Money: startup costs, ongoing costs?
   Attention: what gets displaced if this enters the OS?
   Energy: is the current bandwidth available?

TEST 4 — UNIQUENESS QUESTION
What does Morph specifically bring to this
that someone else cannot replicate easily?
If the answer is nothing — reconsider.
If the answer is clear — that is the foundation.

TEST 5 — THE TIMING QUESTION
Why now?
Is now actually the right time for this?
The 2027 threshold is the current strategic frame —
does this idea serve the pre-threshold build
or does it distract from it?

OUTPUT: Albedo evaluation — each test answered honestly.
        Include the uncomfortable answers.
        A test failed is not automatic rejection —
        it is a design constraint that must be addressed.
```

**ALBEDO GATE:**
Must pass at least 4 of 5 tests to proceed.
Test 3 (Resource Reality) is non-negotiable — 
it must be answered clearly regardless of other outcomes.
If resource reality is unclear, that becomes Step 3's first task.

---

### STEP 3 — CITRINITAS: ILLUMINATION
**Design the project clearly enough to execute.**

An idea that survived purification is now a project candidate. This step builds the blueprint — not the full project plan, but enough clarity that execution has a target.

```
ACTION — Build the project blueprint:

1. PROJECT NAME AND CODE
   A name that is clear and a department code for filing.
   Format: [DEPT-CODE]_[project-name]
   Example: DCE_sovereign-science-series
             DSE_wholesale-buyer-network

2. MISSION STATEMENT (one sentence)
   What is this project trying to accomplish?
   Specific, measurable where possible.

3. SUCCESS DEFINITION
   What does done look like?
   What does successful look like?
   These are often different — name both.

4. PHASE ARCHITECTURE
   Break it into phases. Maximum three phases initially.
   Each phase: what gets built, what it produces, when it ends.
   Do not plan beyond Phase 3 — the project will teach you
   what Phase 4 needs to be once you are inside it.

5. MINIMUM VIABLE VERSION
   What is the smallest version of this that could exist
   and still test whether the idea works?
   This is where execution begins — not at full vision.

6. RESOURCE ALLOCATION
   Based on Step 2's resource reality:
   - Hours per week committed
   - Budget allocated (even if zero)
   - Which department manages it
   - Which other departments it will draw from

7. DEPENDENCIES
   What does this project need from other departments?
   Research from D.R.D?
   Content from D.C.E?
   Financial structure from D.S.E?
   Map those now so they can be requested properly.

8. KILL CONDITIONS
   Under what circumstances does this project get stopped?
   Naming these in advance removes the emotional difficulty
   of stopping something that is not working.
   Example: "If no revenue in 90 days of active operation"
            "If it consistently requires more than X hours/week"

OUTPUT: Project blueprint document.
        This becomes the project's master reference file.
```

**CITRINITAS GATE:**
Can the project be executed with current resources?
If the minimum viable version still requires resources
that do not exist — the project waits until they do,
or the minimum viable version is reduced further.

---

### STEP 4 — RUBEDO: INTEGRATION
**Bring the project into the OS officially.**

A project that has survived all three stages is sanctioned. This step integrates it into the OS so it has a home, receives attention, and connects to the departments it needs.

```
ACTION — Integration sequence:

1. CREATE THE PROJECT FILE
   File: [dept-code]_project_[name]_v1.md
   Contents: The completed blueprint from Step 3
   Location: In the primary department's folder

2. UPDATE CONTEXT FILES
   Add the project to the managing department's context file
   Add a reference in every department the project draws from

3. SET THE FIRST MILESTONE
   From the Phase 1 architecture — what is the
   single first deliverable?
   Name it. Assign it. Put a date on it.
   A project without a first milestone is an intention, not a project.

4. ASSIGN THE ALCHEMICAL STATUS
   Every active project carries a current stage marker:
   🔲 NIGREDO — raw, being evaluated
   ⬜ ALBEDO — being purified / tested
   🔶 CITRINITAS — being designed / planned
   🔴 RUBEDO — active / in execution
   ✅ COMPLETE — finished
   ❌ DISSOLVED — stopped (with kill condition documented)

5. LOG IN DSC MASTER PROJECT REGISTER
   Every sanctioned project is listed in:
   dsc_project_register.md
   With: name, department, stage, first milestone, date entered

OUTPUT: Active project file created, context files updated,
        first milestone set, project register updated.
```

## FAILURE MODES TO AVOID

```
SKIPPING NIGREDO
Starting evaluation before the idea is clearly stated.
Result: You purify something you never actually defined.

ROMANTIC ALBEDO
Passing the purification tests because the idea is exciting
rather than because it actually passed.
The tests are honest or they are useless.

PLANNING WITHOUT MINIMUM VIABLE
Designing the full vision in Citrinitas
without identifying the minimum viable version first.
Result: The project never starts because
the full version is always "not ready."

INTEGRATING WITHOUT A FIRST MILESTONE
Rubedo without a concrete next action.
Result: A sanctioned project that never moves.

NEVER DISSOLVING
Refusing to invoke the kill conditions when they are met.
Result: Dead projects occupying OS attention
and blocking new ones from entering.
```
