# WF_DSC — STAGE 3: CITRINITAS
**Illumination — Design the project clearly enough to execute.**

---

## WHAT THIS STAGE IS

An idea that survived purification is now a project candidate. This stage builds the blueprint — not the full project plan, but enough clarity that execution has a real target. Load `skill_system-design.md` if the architecture is complex.

---

## ACTION — Build the Project Blueprint

**1. PROJECT NAME AND CODE**
A name that is clear and a department code for filing.
Format: `[DEPT-CODE]_[project-name]`
Example: `DCE_sovereign-science-series` / `DSE_wholesale-buyer-network`

**2. MISSION STATEMENT** *(one sentence)*
What is this project trying to accomplish? Specific and measurable where possible.

**3. SUCCESS DEFINITION**
What does *done* look like? What does *successful* look like? These are often different — name both.

**4. PHASE ARCHITECTURE**
Break it into phases. Maximum three phases initially. Each phase: what gets built, what it produces, when it ends. Do not plan beyond Phase 3 — the project will teach you what Phase 4 needs to be once inside it.

**5. MINIMUM VIABLE VERSION**
What is the smallest version of this that could exist and still test whether the idea works? Execution begins here — not at full vision.

**6. RESOURCE ALLOCATION**
- Hours per week committed
- Budget allocated (even if zero)
- Which department manages it
- Which other departments it draws from

**7. DEPENDENCIES**
What does this project need from other departments?
- Research from D.R.D?
- Content from D.C.E?
- Financial structure from D.S.E?
Map these now so they can be requested properly.

**8. KILL CONDITIONS**
Under what circumstances does this project stop? Naming these in advance removes the emotional difficulty of stopping something not working.
> Example: "No revenue in 90 days of active operation" / "Consistently requires more than X hours/week"

---

## OUTPUT

A project blueprint document — this becomes the project's master reference file.

File as: `dsc_project_[name]_citrinitas.md`

---

## GATE

> Can this project be executed with current resources?
> If the minimum viable version still requires resources that do not exist — the project waits, or the minimum viable version is reduced further.

**PASS → load `wf_stage-4_rubedo.md`**
**NOT READY → reduce the minimum viable version further or identify what must be acquired first.**
