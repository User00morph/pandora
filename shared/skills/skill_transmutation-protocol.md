# SKILL — TRANSMUTATION PROTOCOL
**Loadable by: D.S.C, D.S.E, D.P.S.A**

---

## WHAT THIS SKILL IS

The process for converting breakdowns, failures, and underperformance into refined design intelligence. There is no failure in the OS — only unrefined input that reveals what better output requires. Load this skill whenever a system, project, process, or pattern breaks down and needs to be transformed rather than abandoned.

This is the Transmutation Doctrine made executable.

---

## THE PROCESS

```
1. RECEIVE THE BREAKDOWN
   Do not dismiss, minimize, or rush past it.
   The breakdown is information — possibly the most valuable
   information the system has produced.
   Document exactly what happened:
   - What was the expected output?
   - What was the actual output?
   - When did the gap become visible?
   - What was the immediate impact?
   Rule: No blame. No narrative. Just observation.

2. IDENTIFY THE DISTORTION POINT
   Where in the system did the misalignment enter?
   Trace backward from the failure to its origin:
   - Was it in the INTENTION? (wrong goal, unclear purpose)
   - Was it in the DESIGN? (missing component, bad logic)
   - Was it in the EXECUTION? (correct design, poor implementation)
   - Was it in the DATA? (garbage in, garbage out)
   - Was it in the TIMING? (right system, wrong moment)
   - Was it in the ENVIRONMENT? (external factor the system didn't account for)
   Usually the distortion point is upstream of where the failure appeared.

3. EXTRACT THE REFINEMENT
   What does this breakdown reveal about what the system actually needs?
   For each distortion point identified:
   - What was missing that would have prevented this?
   - What assumption was wrong?
   - What was the system blind to?
   The refinement is the gold inside the breakdown.
   State it as a design requirement, not a complaint.

4. MAP THE UPSTREAM FIX
   Apply the edit-source principle:
   Do not fix the output — fix the source that produced it.
   - If the intention was wrong → return to Nigredo
   - If the design was flawed → return to Albedo or Citrinitas
   - If the execution failed → fix the process, not just the result
   - If the data was bad → fix the input pipeline
   Recurring fixes to the same output = diagnostic info
   about which source file needs to change.

5. REDESIGN
   Return to the appropriate alchemical stage.
   Apply the extracted refinement.
   Build the upgraded version with the new constraint built in.
   The upgraded version should make the original failure
   structurally impossible, not just unlikely.

6. TEST THE UPGRADE
   Run the redesigned system through the specific conditions
   that caused the original failure.
   Does it handle them now? Verify directly.
   If it fails again — the refinement extraction was incomplete.
   Return to Step 3.

7. DOCUMENT THE FULL CYCLE
   Record the complete transmutation:
   - What broke (Step 1)
   - Why it broke (Step 2)
   - What was learned (Step 3)
   - What was changed (Step 5)
   - Whether the fix holds (Step 6)
   The breakdown is now an asset in the system's record.
   Future builders learn from this documentation.
```

---

## QUALITY CHECK

Before a transmutation is considered complete:

```
[ ] The breakdown was documented without narrative or blame
[ ] The distortion point was traced upstream, not just at the failure surface
[ ] The refinement is stated as a design requirement
[ ] The fix targets the source, not the output
[ ] The redesign was tested against the original failure conditions
[ ] The full cycle is documented as a permanent record
```
