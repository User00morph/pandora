# SKILL — SYSTEMS CULTIVATION
**Loadable by: D.S.C**

---

## WHAT THIS SKILL IS

The ongoing practice of tending live systems so they grow, adapt, and compound over time rather than stagnate and decay. Systems are not built and abandoned — they are living organisms that require cultivation. This skill governs what happens after Rubedo — the long game of keeping systems healthy, pruning what no longer serves, and recognizing when a system has outgrown its original architecture.

Load this skill during periodic system reviews, when a system shows signs of strain, or when the OS itself needs a health check.

---

## THE PROCESS

```
1. SYSTEM HEALTH SCAN
   For the system under review, assess each dimension:
   
   VITALITY — Is this system still producing its intended output?
   Rate: THRIVING / FUNCTIONING / STAGNANT / DECLINING
   
   ALIGNMENT — Does this system still serve the Sovereign's
   current mission? Missions evolve. Systems that served
   the mission 6 months ago may not serve it now.
   Rate: ALIGNED / DRIFTING / MISALIGNED
   
   EFFICIENCY — Is the ratio of input effort to output value
   improving, stable, or degrading over time?
   Rate: IMPROVING / STABLE / DEGRADING
   
   LOAD — Is this system operating within its designed capacity
   or is it under strain?
   Rate: UNDER CAPACITY / OPTIMAL / STRAINED / OVERLOADED

2. GROWTH PATTERN ASSESSMENT
   How has this system changed since deployment?
   - What has been added that was not in the original design?
   - What has been removed or fallen out of use?
   - What workarounds exist that were not planned?
   Workarounds are diagnostic — they reveal what the
   original design did not account for.
   Document each workaround and what it compensates for.

3. PRUNING AUDIT
   What in this system no longer serves?
   - Components that are maintained but produce no value
   - Processes that run out of habit rather than necessity
   - Outputs that no one uses or acts on
   - Connections to other systems that are no longer active
   Everything identified here is a candidate for removal.
   Pruning is not failure — it is maintenance.

4. COMPOUNDING ASSESSMENT
   Is this system compounding — getting more valuable over time?
   Signs of compounding:
   - Each cycle produces better output than the last
   - The system generates its own inputs (self-feeding loops)
   - Knowledge accumulates and improves future cycles
   - Downstream systems benefit more with each iteration
   Signs of decay:
   - Output quality is flat or declining
   - More effort is required for the same result
   - The system consumes more attention than it produces value
   If decaying — route to transmutation protocol.

5. ADAPTATION PLAN
   Based on the scan, what changes does this system need?
   
   MINOR — Small adjustments within existing architecture
   (parameter changes, process tweaks, component updates)
   Action: Make the changes and document them.
   
   MODERATE — Significant updates that change how parts
   of the system work but not the overall architecture
   Action: Design the updates, test, then implement.
   
   MAJOR — The system has outgrown its architecture
   and needs a fundamental redesign
   Action: Route to system design skill for a new version.
   
   RETIREMENT — The system no longer serves any mission
   and should be gracefully shut down
   Action: Archive everything, update all references,
   and remove from active systems tracker.

6. CULTIVATION SCHEDULE
   Set the next review date for this system.
   - THRIVING systems: review quarterly
   - FUNCTIONING systems: review monthly
   - STAGNANT or DECLINING systems: review in 2 weeks
     or route to transmutation immediately
   Log the schedule in the system's file and
   the D.S.C active systems tracker.
```

---

## QUALITY CHECK

Before a cultivation review is considered complete:

```
[ ] All four health dimensions scored honestly
[ ] Workarounds identified and traced to design gaps
[ ] Pruning candidates named without sentimentality
[ ] Compounding assessment distinguishes growth from maintenance
[ ] Adaptation plan matches the severity of findings
[ ] Next review date is set and logged
```
