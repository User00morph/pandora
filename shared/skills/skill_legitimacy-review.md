# SKILL — LEGITIMACY REVIEW
**Loadable by: D.S.C**

---

## WHAT THIS SKILL IS

The quality gate that determines whether a system or project is ready for deployment into the OS or the physical world. This is not bureaucracy — it is the final check that separates systems that will serve from systems that will create problems. Load this skill before any system goes live, before any project enters active execution, and during periodic audits of running systems.

---

## THE PROCESS

```
1. INTENTION CLARITY AUDIT
   Is the purpose of this system clearly defined?
   Can it be stated in one sentence without ambiguity?
   Does the stated purpose match what the system actually does?
   If the intention has drifted from original design — document why.
   Score: CLEAR / VAGUE / DRIFTED

2. ARCHITECTURAL COMPLETENESS CHECK
   Are all components identified and documented?
   Map against the system design skill output:
   - Inputs: all identified? ☐
   - Processes: all steps mapped with input → action → output? ☐
   - Outputs: all defined with quality standards? ☐
   - Feedback loop: exists and is measurable? ☐
   Any missing component = the system is incomplete.
   Score: COMPLETE / GAPS IDENTIFIED / INCOMPLETE

3. ALCHEMICAL STAGE CONFIRMATION
   Has the system completed Nigredo through Citrinitas?
   - Nigredo: raw material was received and documented? ☐
   - Albedo: purification tests were run honestly? ☐
   - Citrinitas: design is clear enough to execute? ☐
   A system that skipped stages will fail in ways
   that could have been caught earlier.
   Score: ALL STAGES COMPLETED / STAGES SKIPPED

4. DEPARTMENT ALIGNMENT VERIFICATION
   Does this system belong in the department it is assigned to?
   Does it integrate cleanly with existing systems in that department?
   Does it conflict with or duplicate any existing system?
   Are cross-department connections mapped and handoffs defined?
   Score: ALIGNED / MISALIGNED / CONFLICTS EXIST

5. TRANSMUTATION READINESS
   If this system breaks — is there a protocol for what happens next?
   Has at least one failure mode been identified?
   Is there a kill condition defined?
   Is there a person or process responsible for monitoring?
   A system without failure handling is a system waiting to fail silently.
   Score: READY / PARTIALLY READY / NOT ADDRESSED

6. RESOURCE SUSTAINABILITY
   Can this system be maintained with current resources?
   Does it require ongoing attention, and if so, how much?
   Is it self-sustaining or does it depend on manual intervention?
   What happens if Morph is unavailable for a period?
   Score: SUSTAINABLE / ATTENTION-DEPENDENT / UNSUSTAINABLE

7. FINAL VERDICT
   Based on all six checks:
   DEPLOY → system passes all prerequisites, ready to go live
   REMEDIATE → specific gaps identified, return to the appropriate stage
   REJECT → fundamental issues that cannot be remediated without redesign
```

---

## QUALITY CHECK

Before a legitimacy review is considered complete:

```
[ ] All six checks have been run with honest scoring
[ ] No check was skipped because the answer seemed obvious
[ ] Any score below the top rating has a specific remediation path
[ ] The verdict is documented with reasoning
[ ] If REMEDIATE — the specific stage to return to is named
[ ] If DEPLOY — deployment date and monitoring plan are set
```
