# Protocol: Development Railway
## D.S.C → D.I.I | All Development Initiatives, Repos & Agentic Builds

Every piece of development work — incoming repos, software builds, agentic systems, MCP servers, APIs, VS Code integrations — moves through this railway. No exceptions. D.S.C is the spine. D.I.I is the execution layer.

---

## THE RAILWAY

```
STAGE 0 — SIGNAL
  A repo is found, a build idea surfaces, an external tool is identified
  Entry point: D.S.C

STAGE 1 — INTAKE (D.S.C)
  Evaluate the signal. Does it serve Pandora OS? Which department benefits?
  Files: wf_trigger.md + wf_stage-1_nigredo.md
  Output: intake decision — proceed or discard

STAGE 2 — PRD (D.S.C)
  Write the build contract before any execution begins.
  Files: skill_prd-creation.md
  Output: dsc_prd_[project-name]_v1.md — approved by Morph before Stage 3

STAGE 3 — EXTRACTION & EVALUATION (D.I.I)
  Pull the repo. Evaluate architecture, license, resource requirements, fit.
  Files: skill_repo-extraction.md + skill_source-evaluation.md
  Output: evaluation summary filed in D.I.I or D.R.D

STAGE 4 — BUILD & INTEGRATION (D.I.I)
  Integrate into Pandora agentic layer. Wire to VS Code, Claude Code, or MCP layer.
  Files: skill_agentic-architecture.md + skill_software-build-protocol.md
  Output: working integration or agent definition

STAGE 5 — TRACKING (D.S.C)
  Log deployed integration in D.S.C ACTIVE STATE.
  Update D.I.I memory core with new tool stack entry.
  Update D.I.I hardware sovereignty roadmap repo queue.
```

---

## ROUTING RULES

| Situation | Route |
|-----------|-------|
| New repo discovered | D.S.C intake first — always |
| Build idea (software, MCP, API) | D.S.C PRD first — always |
| Research needed before PRD | D.R.D → return to D.S.C |
| Repo has revenue application | D.S.E notified after D.S.C PRD |
| Repo generates content output | D.C.E notified after D.I.I integration |
| Tech-magik convergence in repo | D.O.M flagged during D.I.I evaluation |
| Build is complete | D.S.C ACTIVE STATE updated + D.I.I memory core updated |

---

## REPO EVALUATION CRITERIA (D.I.I Stage 3)

When extracting and evaluating an incoming repo, assess against these in order:

1. **License** — MIT, Apache 2.0, or equivalent. No GPL if distribution is intended.
2. **Resource fit** — does it run on current hardware or Phase 2 target hardware?
3. **Interface** — does it connect to VS Code, Claude Code CLI, or Pandora file system?
4. **Agentic capability** — tool use, file access, multi-step reasoning support?
5. **Open weights / no vendor lock** — no dependency on closed APIs if sovereignty is the goal
6. **Maintenance health** — active commits, community size, issue response rate

## WHAT NEVER SKIPS STAGE 2 (PRD)

- Any software build involving Claude Code execution
- Any MCP server or API integration
- Any repo that writes to Pandora file system
- Any agentic system with autonomous execution capability

The PRD is the contract. No execution before Morph approves it.
